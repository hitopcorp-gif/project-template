# 開発絶対ルール — 田中貴大プロジェクト共通テンプレート

> **このファイルはすべての作業の最上位ルールである。**
> 新規プロジェクト開始時、このファイルの全項目を確認してから作業を開始すること。
> チャット圧縮後・新セッション開始時にも必ず再読すること。

---

## 0. このファイルの位置づけ

- 機能追加・バグ修正・リファクタリング、いかなる作業の前にもこのルールに違反しないことを確認する
- 「動けばいい」は禁止。センス・クオリティ・ディテールの基準を一切妥協しない

### AI への指示

1. **新規プロジェクト開始時:** 全項目を読んでから作業開始。読み飛ばし禁止
2. **新しいセッション開始時:** 最初にこのファイルを再読
3. **チャット圧縮後:** このファイルを読めば全ルールを復元できる
4. **push 前:** CI を確認し、緑になるまで修正。たかひろさんの手作業はゼロ

---

## 1. Fat Component の絶対禁止

| 対象 | 上限 (行) | 備考 |
|---|---|---|
| ページ (`src/pages/**`) | 600 | 超過時は即分割 |
| コンポーネント (`src/components/**`) | 400 | |
| フック (`src/hooks/**`) | 300 | |
| ユーティリティ (`src/utils/**`) | 250 | |
| テスト (`**/__tests__/**`) | 500 | |
| 型定義 (`src/types/**`) | 400 | |

**300 行を超えたら危険信号。技術的理由の文書化がない限りリファクタリング必須。**

### 関心事の分離

- `src/components/` — UI 描画のみ
- `src/hooks/` — データ取得・状態管理
- `src/utils/` — React 非依存の純粋関数

---

## 2. CI/CD 品質ゲート (`.github/workflows/quality-gate.yml`)

以下を push / PR ごとに自動チェック:

1. **ファイル行数制限** — ARCHITECTURE.md 準拠
2. **`any` 型の禁止** — テスト・モック除く
3. **`console.log/info` の禁止** — 本番コード内
4. **ハードコード API キーの検出** — `AIza...`, `sk-...`, `AKIA...`
5. **テストファイル存在チェック** — utils/hooks は必須
6. **テスト実行** — 全パス必須
7. **浮動小数点金額計算の検出** — 整数(円) + `Math.floor` を使用
8. **TypeScript 型チェック** — `tsc --noEmit`
9. **ビルド成功**

### AI の責務

- push 後に CI 結果を確認する
- 赤なら即修正して再 push
- **緑になるまでセッションを終了しない**
- 完了報告には必ず「CI: ✅ 緑」を含める

---

## 3. データベース設計

- **スナップショット設計:** データ生成時、その時点のマスタ情報を焼き付ける
- **テナントID:** マルチテナント対応時は全テーブルに `tenantId`
- **ビジネスルールのハードコード禁止:** 設定ファイルまたは DB に外出し

> ※ プロジェクトごとに DB サービスを選定:
> Firebase Firestore / Cloudflare D1 / Supabase 等

---

## 4. 金額計算

- **整数 (円) のみ使用** — 浮動小数点禁止
- **税額丸め:** デフォルト `Math.floor`
- **明細保持:** 検算可能な内訳を必ず保存

---

## 5. UI デザイン

- **モバイルファースト:** タップターゲット >= 44px
- **Tailwind CSS** 使用
- **3 ブレークポイント:** >=1024px / 768-1024px / <=768px
- **`@media print` 対応** — A4 印刷レイアウト（必要な場合）
- **管理者画面:** 白黒基調、テーブル形式、装飾排除

> ※ プロジェクトごとにデザインガイドを `PROJECT.md` に追記すること

---

## 6. セキュリティ

- **フロントエンドに API キーを置かない** — サーバーサイド (Workers / Functions) でプロキシ
- **認証・ロールチェック必須** — 全 API エンドポイント
- **DB セキュリティルール設定** — Firestore Rules / Supabase RLS / D1 Auth

---

## 7. テスト

- **新規・変更ソース → テスト自動作成**
- **命名:** `src/utils/calc.ts` → `src/utils/__tests__/calc.test.ts`
- **カバレッジ基準:**
  - `src/utils/` — 100%
  - `src/hooks/` — >= 80%
- **エッジケース必須:** 0, 負数, 巨大数, 丸め誤差

---

## 8. コード分割 & ビルド

- `React.lazy` + `Suspense` でルート分割
- `manualChunks` で大きなライブラリを分離
- 本番ビルドで `console.log` / `debugger` をストリップ

---

## 9. 外部サービスパターン

- **HTTP 通信:** `fetchWithRetry` (タイムアウト・リトライ標準化)
- **API キー管理:** `.dev.vars` (ローカル) + `wrangler secret put` (本番)

> ※ プロジェクト固有の外部サービス (OCR, AI, 決済等) は `PROJECT.md` に記載

---

## 10. Wave プロセス

**3 機能以上または大型機能の後、以下のチェックリストを実行:**

1. RULES.md 再読
2. ファイル行数チェック (`npm run check:filesize`)
3. テスト作成・実行 (`npm test`)
4. TypeScript チェック (`tsc --noEmit`)
5. 重複コード検出
6. `console.log` / `debugger` 除去
7. バンドルサイズチェック (`npm run build`)
8. セキュリティ監査 (API キー、環境変数)

**コミットメッセージに `[Wave X]` を含める** (X はアルファベット順: A, B, C...)

### コミットメッセージルール

- **what と why を記述** — `fix` や `update` だけは禁止
- 例: `feat: NumericInput で value=0 の入力不具合を解消 — blur 時にクランプ処理追加`

## 12. プロジェクト固有設定 (`PROJECT.md`)

各プロジェクトのルートに `PROJECT.md` を作成し、以下を記載:

- プロジェクト名・概要
- DB サービスと構造
- 外部サービス連携
- UI デザインガイド
- 技術スタックの追加・変更
- 既知の技術的負債リスト
- 会社情報（請求書等に記載する場合）
